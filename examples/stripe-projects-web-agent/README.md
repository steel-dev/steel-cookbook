# Web Agent Studio with Stripe Projects

Stripe Projects provisions this Next.js application with Steel browser infrastructure, an OpenRouter model, and a Vercel project. The agent opens managed browser sessions, navigates public webpages, inspects page content, and returns a structured report with source links. The interface streams its tool activity and embeds each active session in Steel Live View.

Each run is bounded to 12 agent steps and four concurrent browser sessions. The Stop control cancels the active stream, and the request lifecycle releases its Steel sessions when work finishes or stops.

## Build and run

Create the application through Stripe Projects:

```bash
stripe plugin install projects
stripe projects build my-agent --template steel/web-agent-studio
cd my-agent
npm run dev
```

Open [http://localhost:3000](http://localhost:3000), choose a sample task or write your own, and start the run. The browser panel appears when the agent opens its first Steel session.

The build command materializes the pinned template, installs its dependencies, provisions all three services, and writes their configuration into the project. This example has no manual provider-key setup path.

## Provisioned stack

| Service | Role in the application |
| --- | --- |
| Steel | Creates managed browser sessions and provides embedded Live View URLs |
| OpenRouter | Runs the model behind the server-side agent loop |
| Vercel | Provides the project targeted by the deployment script |

The agent runtime reads two generated variables:

```text
STEEL_API_KEY
OPENROUTER_API_API_KEY
```

The repeated `API` in `OPENROUTER_API_API_KEY` is intentional. It matches the environment contract generated by Stripe Projects. The API route validates both variables before starting a run.

## Agent flow

The browser client posts a task to `/api/chat`. The route streams model output and tool results back to the page while keeping browser state on the server.

The model can call four tools:

- `openSession` creates a Steel session and returns the Live View metadata used by the UI.
- `navigate` loads a public HTTP or HTTPS URL in a selected session.
- `inspectPage` returns the current page title, URL, text, and links for model context.
- `deliverReport` returns typed findings and source links for the final report card.

`openSession` can create up to four sessions in one run, which lets the agent compare sources without an unbounded browser fan-out. The server forces the first browser step, keeps the loop on tool calls, and forces `deliverReport` on the final allowed step. The run ends no later than step 12.

The UI renders each tool transition in the activity timeline. Active sessions appear as Live View tabs, and the completed response appears as a structured report instead of an unformatted chat message.

## Deploy the provisioned project

The template includes a deployment helper for the Vercel project created by Stripe Projects:

```bash
npm run deploy
```

The helper loads the Stripe-generated local environment, syncs only `STEEL_API_KEY` and `OPENROUTER_API_API_KEY` to Vercel production, uploads the application, and starts a production deployment. It excludes `.env` files, the local `.projects` workspace, and Vercel control credentials from the upload.

If the Vercel credentials are missing or stale, refresh the project environment and retry:

```bash
stripe projects env --pull
npm run deploy
```

The deployed route uses your provisioned Steel and OpenRouter resources. Add authentication and per-user rate limits before sharing it as a public application.

The route rejects literal local and private destinations and screens DNS before browser requests. That resolution runs in the application while the Steel browser connects from its own network, so treat the guard as defense in depth rather than a private-network egress boundary. Enforce private-network restrictions at the browser or proxy layer before accepting untrusted destinations in a public deployment.

Run `npm test` after changing the URL guard. Its focused suite covers successful cache reuse, expiry, concurrent resolution, failure retries, and the stale-lookup race.

## Make it yours

- Change the seeded tasks in the page component to match the workflows you want to demonstrate.
- Adjust the system prompt and step limit in the chat route for a narrower agent policy.
- Extend `inspectPage` with application-specific extraction fields.
- Add tools for screenshots, downloads, or authenticated workflows while preserving session cleanup.
- Extend the report schema when the application needs richer findings than a title, summary, and source list.

Keep provider credentials on the server. Client components should consume streamed tool results and Live View metadata, never the generated API keys.
